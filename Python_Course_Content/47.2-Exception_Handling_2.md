# ✅ **Lesson: Python Exceptions & Custom Exceptions (Practical, Example-Driven)**

Exceptions are central to writing robust, maintainable Python programs. This lesson walks through how exceptions work, how to handle them, how to create custom exception types, and how to chain exceptions to preserve debugging context.

***

# 1. ✅ **Built‑In Exceptions**

Python automatically raises exceptions when something unexpected happens (e.g., invalid input, missing files, type mismatches). These are known as **built-in exceptions** such as:

*   `ValueError`
*   `TypeError`
*   `KeyError`
*   `FileNotFoundError`
*   `ZeroDivisionError`
*   `IndexError`
*   `TimeoutError`
*   … and many more

You can see the full hierarchy with `help(Exception)` or in the Python docs.

***

## ✅ **Example 1 — Handling Built‑In Exceptions with `try`, `except`, `else`, and `finally`**

```python
def load_config(path):
    try:
        # Try reading a file that may not exist
        with open(path, "r") as f:
            config = f.read()

    except FileNotFoundError:
        # Handle a specific built-in exception
        print(f"Error: The file '{path}' was not found.")
        return {}

    except PermissionError:
        # Another common file-related exception
        print(f"Error: No permission to read '{path}'.")
        return {}

    else:
        # Runs only if NO exceptions were raised
        print("Config loaded successfully.")
        return config

    finally:
        # Always runs, exception or not
        print("Attempted to load configuration file.")
```

### ✅ Explanation

*   `try`: Code that *might* fail.
*   `except`: What to do if the specific error occurs.
*   `else`: Executes only if the try block succeeds—great for success-path logic.
*   `finally`: Executes regardless of success or failure—commonly used for cleanup.

***

# 2. ✅ **Catching Specific vs. Broad Exceptions**

Catching a broad exception like `Exception` hides bugs and makes debugging harder.

***

## ✅ **Example 2 — Why catching specific exceptions matters**

```python
def parse_int(value):
    try:
        return int(value)   # Might raise ValueError or TypeError

    except ValueError:
        print(f"Could not convert '{value}' to an integer.")
        return None

    except TypeError:
        print("Received a value that cannot be converted (non-string, non-numeric).")
        return None
```

### ✅ Explanation

Catching exceptions *one by one* ensures you handle exactly what you intend.

Now compare with this **dangerous anti-pattern**:

```python
def parse_int_dangerous(value):
    try:
        return int(value)
    except Exception:
        # This catches EVERYTHING—including bugs you weren't expecting.
        print("Something went wrong.")
        return None
```

### ✅ Why this matters

*   Overly broad `except` hides real programming errors.
*   You may silence unexpected issues like `KeyboardInterrupt`, `MemoryError`, or logic bugs.
*   Good exception handling is **precise**, not generic.

***

# 3. ✅ **Raising Exceptions Manually with `raise`**

You can trigger exceptions intentionally when your program detects invalid state.

***

## ✅ **Example 3 — Manual validation in a data-processing pipeline**

```python
def validate_age(age):
    if age is None:
        raise ValueError("Age is required.")

    if not isinstance(age, int):
        raise TypeError(f"Age must be an integer, not {type(age).__name__}")

    if age < 0:
        raise ValueError("Age cannot be negative.")

    return age
```

### ✅ Explanation

Using `raise`:

*   Helps you enforce business rules
*   Prevents corrupted or invalid data from propagating
*   Fails early with clear messages

# 4. ✅ **Custom Exceptions**

Python allows you to define your own exception classes by subclassing `Exception`.

### ✅ When to use custom exceptions

Use custom exceptions when:

*   A built-in exception doesn't clearly describe the issue
*   You want users of your module to catch domain-specific problems
*   You want to attach extra context (fields, error codes, metadata)

***

# 4.1 ✅ **Example 4 — Basic Custom Exception**

```python
class InvalidOrderState(Exception):
    """Raised when an order transitions into an invalid state."""
    pass


def update_order_status(order_id, new_status):
    valid_statuses = {"PENDING", "PAID", "SHIPPED"}

    if new_status not in valid_statuses:
        raise InvalidOrderState(
            f"Order {order_id}: Cannot set status to '{new_status}'."
        )

    print(f"Order {order_id} updated to {new_status}.")
```

### ✅ Explanation

*   Inherits from `Exception`
*   Contains a clear purpose and message
*   Helps distinguish this kind of error from general failures

***

# 4.2 ✅ **Example 5 — Custom Exceptions with Attributes**

```python
class APIError(Exception):
    """Represents an error returned from a remote API."""

    def __init__(self, status_code, message, endpoint):
        super().__init__(message)
        self.status_code = status_code
        self.endpoint = endpoint


def fetch_data(endpoint):
    # Simulate an API failure
    fake_status_code = 503

    if fake_status_code != 200:
        raise APIError(
            status_code=fake_status_code,
            message="Service Unavailable",
            endpoint=endpoint
        )
```

### ✅ Explanation

This exception:

*   Stores metadata (`status_code`, `endpoint`)
*   Makes debugging easier
*   Allows callers to inspect error details programmatically

# 5. ✅ **Exception Chaining with `raise ... from ...`**

Sometimes you want to raise a new exception but keep the original cause.

Python supports this using `raise NewError(...) from original_error`.

***

## ✅ **Example 6 — Exception chaining in data parsing**

```python
import json

class ConfigParseError(Exception):
    """Raised when configuration JSON cannot be parsed or used."""

def load_json_config(path):
    try:
        with open(path, "r") as f:
            raw = f.read()
            return json.loads(raw)

    except (FileNotFoundError, PermissionError) as file_err:
        # Preserve root cause: missing file or permissions
        raise ConfigParseError(
            f"Could not load configuration from {path}"
        ) from file_err

    except json.JSONDecodeError as json_err:
        # Preserve root cause: JSON formatting issue
        raise ConfigParseError(
            f"Invalid JSON in configuration file: {path}"
        ) from json_err
```

### ✅ Explanation

Exception chaining:

*   Provides a clean, domain-specific error to the caller
*   Still preserves original traceback for debugging
*   Avoids losing root causes, which is crucial in production environments

***

# ✅ **Summary: Best Practices for Exception Handling in Python**

### ✅ **1. Prefer specific exception types**

Avoid catching broad exceptions unless absolutely necessary.

### ✅ **2. Use `try/except/else/finally` intentionally**

*   Use `else` for success-path logic
*   Use `finally` for cleanup (closing files, releasing locks, etc.)

### ✅ **3. Raise exceptions early to enforce contracts**

Don’t let invalid data propagate.

### ✅ **4. Create custom exceptions for domain-specific errors**

Especially in libraries, APIs, or business logic.

### ✅ **5. Add attributes to exceptions when context is helpful**

Status codes, filenames, query details, etc.

### ✅ **6. Use exception chaining (`raise ... from ...`)**

Preserves original error context—hugely valuable for debugging.

### ✅ **7. Avoid using bare `except:`**

This catches too much, including `KeyboardInterrupt` and `SystemExit`.

### ✅ **8. When in doubt: Fail loudly, fail clearly.**

Clear exceptions save hours of debugging later.
